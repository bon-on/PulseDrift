# 제약

- 기본 정책 모드는 드리프트는 경고하고, 치명적 공백은 차단한다.
- 모든 기능 변경은 `harness/specs/` 안의 기능 명세를 먼저 가져야 한다.
- 아키텍처 또는 횡단 관심사 결정은 `harness/adrs/` 안의 ADR이 필요하다.
- AI 응답은 사용한 명세, ADR, 규칙을 반드시 인용해야 한다.
- 저장소 상태가 명세나 ADR과 충돌하면 추측하지 말고 에스컬레이션한다.
- 운영 앱은 공유 Flutter 코드베이스에서 iPhone과 Android를 모두 지원해야 한다.
- 게임은 3레인 회피 구조를 유지하며, 드래그 입력도 결국 이산 레인으로 해석되어야 한다.
- 속도, 스폰 간격, 점수 계산은 UI 상수에 흩어놓지 말고 공유 게임 설정에 둔다.
- 이후 제품 변경은 구현 전에 `harness/specs/`에 Markdown 명세로 기록한다.

## 감지된 스택 요약

- 하네스 기반 전달 방식을 사용하는 Flutter 모바일 게임 저장소

## 제안 프로필 팩 제약

현재 저장소 신호만으로 승인된 프로필 팩은 없다. 스택 지침이 필요하면 규칙 문서에 직접 명시한다.
